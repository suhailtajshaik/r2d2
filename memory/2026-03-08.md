**The PRD Journey — 3 Stages of Prompt Sophistication:**

- **Stage 1: Raw Request** → Just a one-liner, no details
  - Example: "Build me a calorie tracking app"
- **Stage 2: Basic PRD** → Project Requirements Document with feature list
  - Example: "Build a calorie tracking app" with features:
  - • Log food with calories and macros
  - • Assign food to meal categories
  - • Daily dashboard with progress
  - • User authentication
  - • AI analyzes food photos for nutrition
- **Stage 3: Platform-Optimized** → Full tech-stack-aware PRD
  - Example: "Build a calorie tracking app"
  - ✅ React with TypeScript
  - ✅ Tailwind CSS styling
  - ✅ Supabase auth & real-time DB
  - ✅ shadcn/ui components
  - ✅ Calorie/macro charts with Recharts
  - ✅ Responsive mobile-first design

**The Vibe Coding Workflow (in order):**
1. **Research** → Use Perplexity to research what features your app needs (competitive analysis, best practices, user expectations)
2. **PRD** → Then write your PRD based on research (Stage 1 → 2 → 3 progression)
3. **Build** → Feed the PRD to your vibe coding tool

*Key insight: Don't just guess features — research first on Perplexity, THEN write the PRD. The PRD is only as good as the research behind it.*

**The Six Commandments of Vibe Coding:**

1. **Start with a Plan**
   - AI cannot read your mind, only scripts
   - Create clear PRD before coding
   - Use competitor research for comprehensive features
2. **Give Specific, Clear Instructions**
   - ❌ "Build me a recipe app"
   - ✅ "First create a simple homepage with a search bar in the middle and a title at the top that says 'Find your next meal'"
   - When AI misunderstands: Break into smaller, clearer parts
3. **Build and Test in Small Pieces**
   - Add one feature at a time
   - Test after each addition
   - Debug immediately
4. **Save Your Work Constantly**
   - Use version history in Lovable
   - Can restore previous versions
   - Click restore if something breaks badly
5. **Don't Panic, Start Fresh**
   - Avoid sunk cost fallacy
   - If too messy, revert to previous version
   - Better to restart than fix irreparably broken code
**When Things Break — The Debug Loop:**
See Error → Don't Panic → Copy Message → Paste to Bolt (or your AI tool) → Fix or Revert

**Common Errors You'll See:**
- **Module not found** → Missing dependency
- **Undefined variable** → Typo or scope issue
- **DB connection failed** → Check Supabase
- **Blank screen** → Check console

6. **Use Platform Guidelines**
   - Every platform has specific prompting styles
   - Lovable prompting bible example
   - Bolt, Cursor have their own best practices

## Day 3 — Session 2: Automation & n8n (Final Session of Course)

### The Automation Promise vs Reality

**What you've probably heard:**
- "Make money while you sleep!"
- "No coding skills required!"
- "Just drag, drop, and boom - you're an AI agency!"

**The Reality Check:**
- Even developers with experience take a **week** to build something truly useful
- "No-code" doesn't mean "no-logic" or "no-thinking"
- The learning curve is real, but the rewards are astronomical

### The Tutorial Trap — Why Even Smart People Struggle
1. Watch tutorial ✅
2. Follow along ✅
3. Feel like you "get it" ✅
4. Try to build something real ❌
5. Brain draws a blank ❌

**The Problem:** Passive learning feels productive but creates an illusion of knowledge.

### The Science of Learning Automation — What Actually Works:

**1. Active Recall**
1. Build workflows from memory, not tutorials
2. Close the guide and recreate what you just learned
3. Fail, struggle, then succeed — this locks in knowledge

### The Right Mindset — 3 Shifts:

1. **From:** "I want to learn automation" → **To:** "I want to solve this specific problem"
2. **From:** "This should be easy" → **To:** "This is worth the effort"
3. **From:** "I'll watch more tutorials" → **To:** "I'll build and break things"

*Remember: Automation is like compound interest — small improvements add up to massive results.*

### The Planning Framework — NEVER Start Building Without This:

1. **Draw It Out First**
   - Use iPad, notebook, or Whimsical/Miro
   - Map the entire flow before touching any tool

2. **Start with the End**
   - What's the final outcome?
   - What data needs to go where?
   - What actions should happen?

3. **Identify Decision Points**
   - Where do you need IF/THEN logic?
   - What are the failure scenarios?

**Golden Rule: 10 minutes planning saves 50 minutes debugging.**

### Essential Vocabulary (Non-Technical Explanations):

**API (Application Programming Interface)**
1. Think: "Menu for apps"
2. How different software talks to each other
3. You don't need to build APIs, just use them

**Data Types That Matter:**
- **Text/String** — Names, messages, URLs
- **Numbers** — Prices, quantities, IDs
- **Booleans** — Yes/No, True/False
- **Arrays** — Lists of items
- **Objects** — Structured data with multiple fields

**JSON (JavaScript Object Notation)**
- "Digital filing cabinet" — how data is structured between systems
- 99% of modern APIs use JSON
- Understanding structure helps with debugging & data mapping

### Where AI Fits Naturally in Automation:

**AI as a Processing Step:**
1. **Get Data** → 2. **AI Analysis** → 3. **Take Action**

**Common AI Use Cases:**
- **Extract** — Pull specific info from text
- **Classify** — Categorize emails, leads, content
- **Generate** — Create personalized messages
- **Transform** — Convert data formats
- **Decide** — Choose next steps based on criteria

*Remember: AI makes the "Process" step incredibly powerful.*
